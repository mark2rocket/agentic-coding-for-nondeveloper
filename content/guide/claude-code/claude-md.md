CLAUDE.md 파일은 클로드 코드(Claude Code)가 프로젝트를 시작할 때 가장 먼저 읽는 **'프로젝트 헌법'**이자 **'신입 사원 업무 매뉴얼'**입니다.

실무에서 이 파일을 단순한 메모장이 아니라 강력한 시스템 프롬프트로 활용하여 생산성을 극대화하는 방법을 소개합니다.

## CLAUDE.md란?

<Callout type="info" title="업무 매뉴얼로 이해하기">

회사에 신입 사원이 입사하면 **업무 매뉴얼**을 주죠.

"이 업무는 이렇게 처리하세요", "우리 팀 규칙은 이렇습니다"라고 적힌 매뉴얼이 있으면:
- 매번 설명하지 않아도 됩니다
- 일관된 품질로 업무를 수행할 수 있습니다

CLAUDE.md가 바로 클로드 코드에게 주는 **'업무 매뉴얼'**입니다.

</Callout>

**역할:**
- Claude가 세션 시작 시 가장 먼저 읽는 파일
- 프로젝트 규칙, 코딩 컨벤션, 선호사항을 정의
- 매번 같은 설명을 반복하지 않아도 됨

---

## 실무 활용 5가지 핵심 전략

### 1. "무엇(What)"보다 "왜(Why)"를 적으세요

<LevelBox>

단순한 지시보다 이유를 함께 적었을 때 클로드의 판단력이 훨씬 좋아집니다.

**❌ 나쁜 예시:**
```markdown
- TypeScript Strict 모드를 켜.
```

**✅ 좋은 예시:**
```markdown
- 타입 관련 버그를 사전에 방지하기 위해 TypeScript Strict 모드를 사용해.
```

**효과:**
- 이유를 명시하면 클로드가 예상치 못한 상황에서도 사용자의 의도(버그 방지)를 파악하여 더 적절한 코드를 작성합니다.

</LevelBox>

---

### 2. 파일의 성격에 따라 '계층(Scope)'을 나누세요

<LevelBox>

모든 규칙을 하나의 파일에 다 넣으면 토큰 낭비가 심하고 관리가 어렵습니다.

용도에 따라 파일을 분리하여 관리하는 것이 효율적입니다.

</LevelBox>

**파일 계층 구조:**

| 파일명 | 위치 | 용도 및 특징 |
|--------|------|------------|
| `CLAUDE.md` | 프로젝트 루트 | **팀 공용 규칙**<br/>기술 스택, 아키텍처, 코딩 컨벤션 등<br/>팀원 모두가 지켜야 할 약속 (Git 공유) |
| `CLAUDE.local.md` | 프로젝트 루트 | **개인 전용 규칙**<br/>나만의 프롬프트 스타일, 한국어 응답 강제 등<br/>개인적인 선호 사항 (Git Ignore 권장) |
| `~/.claude/CLAUDE.md` | 홈 디렉토리 | **전역(Global) 규칙**<br/>모든 프로젝트에 공통적으로 적용하고 싶은<br/>나의 기본 설정 |
| `하위폴더/CLAUDE.md` | 특정 하위 폴더 | **폴더별 규칙**<br/>해당 폴더(예: `/backend`)에 진입했을 때만<br/>읽히는 특수 규칙 (토큰 절약) |

---

### 3. 프로젝트의 '핵심 문맥'만 담아 '가볍게' 유지하세요

<LevelBox>

클로드는 한 번에 처리할 수 있는 지시사항의 수(약 **150~200개**)에 한계가 있습니다.

파일이 너무 길어지면 일부 규칙이 무시될 수 있습니다.

**포함할 내용:**
- ✅ 프로젝트에 특화된 빌드/테스트 명령어
- ✅ 핵심 아키텍처(MVC, MVVM 등)
- ✅ 네이밍 규칙
- ✅ 자주 발생하는 실수 방지책

**뺄 내용:**
- ❌ "리액트란 무엇인가?"와 같은 일반적인 지식
- ❌ 너무 뻔한 코딩 기초 지식

</LevelBox>

---

### 4. `#` 명령어로 '살아있는 문서'로 만드세요

<LevelBox>

작업 도중 클로드가 같은 실수를 반복하거나, 새로운 규칙이 필요하다고 느껴지면 터미널에서 `#` 키를 사용하여 즉시 규칙을 추가하세요.

**사용법:**

터미널 창에서 `#`를 입력하고:
```
# 모든 주석은 한국어로 작성해
```

**효과:**
- 이 내용이 자동으로 `CLAUDE.md`에 추가됩니다
- 프로젝트를 진행하면서 발견된 노하우가 실시간으로 축적되어, 시간이 갈수록 AI가 더 똑똑해집니다

</LevelBox>

---

### 5. 작업 절차를 강제하는 '트리거'로 활용하세요

<LevelBox>

CLAUDE.md에 코딩 규칙뿐만 아니라 **업무 프로세스**를 정의하여 에이전트의 행동을 제어할 수 있습니다.

**활용 예시:**

```markdown
# 작업 프로세스

## 새 기능 개발 시
1. 먼저 `spec.md` 파일을 읽고 요구사항을 파악한다
2. 기술 설계서 `DESIGN.md`를 작성한다
3. 설계서를 리뷰받은 후 코드 작성을 시작한다
4. 구현 후 반드시 테스트 케이스를 작성한다

## 코드 작성 규칙
- 모든 함수는 JSDoc 주석 필수
- 복잡한 로직은 별도 유틸 함수로 분리
- 상수는 `constants.ts`에 정의
```

**효과:**
- AI가 마음대로 코드를 짜는 것을 방지
- 기획 문서나 기술 명세서를 기반으로 구현하도록 강제
- 일관된 개발 프로세스 유지

</LevelBox>

---

## 4가지 황금 원칙

<Callout type="success" title="CLAUDE.md 작성 원칙">

1. **간결성**: 150~200개 지시사항 제한 (시스템 프롬프트가 ~50개 차지)
2. **특화성**: 프로젝트 고유 내용만 (일반적인 설명 불필요)
3. **이유 포함**: "왜"를 명시하면 AI가 맥락 이해
4. **지속 업데이트**: 반복되는 실수는 즉시 문서화

</Callout>

## 나쁜 예시 vs 좋은 예시

### ❌ 나쁜 예시

```markdown
# 개발 규칙

- TypeScript strict mode 사용
- 깔끔한 코드 작성
- 버그 없이 개발
- 테스트 작성
- 주석 달기
```

**문제점:**
- 이유가 없어서 AI가 왜 그래야 하는지 모름
- 추상적이고 모호한 지시
- 실행 가능한 구체적 가이드 없음

### ✅ 좋은 예시

```markdown
# 개발 규칙

## 타입 안정성
- 타입 관련 런타임 버그를 방지하기 위해 TypeScript strict mode 사용
- `any` 타입 사용 시 반드시 주석으로 이유 명시

## 가독성
- 팀원들이 빠르게 이해할 수 있도록 함수는 20줄 이내로 작성
- 복잡한 로직은 별도 함수로 분리하고 의도를 설명하는 함수명 사용

## 품질 보증
- 프로덕션 환경에서 발생한 버그는 반드시 테스트 케이스 추가 후 수정
- 모든 API 엔드포인트는 에러 처리 포함

## 문서화
- public 함수에는 JSDoc 주석 필수
- 비즈니스 로직이 복잡한 부분은 왜(Why) 중심으로 주석 작성
```

**장점:**
- 이유가 명확해서 AI가 의도를 이해
- 구체적이고 실행 가능한 지시
- 측정 가능한 기준 (예: 20줄 이내)

## 실시간 업데이트 팁

<Callout type="tip" title="# 명령어로 즉시 업데이트">

대화 중 `#` 키를 입력하면 Claude가 CLAUDE.md에 자동으로 지시사항을 추가합니다.

```bash
# 터미널에서 입력
# API 응답은 항상 try-catch로 감싸서 에러 처리해

# → 자동으로 CLAUDE.md에 추가됨
```

</Callout>

## 요약

<Callout type="success" title="CLAUDE.md 실무 활용 요약">

CLAUDE.md를 **팀 전체가 공유하는 코딩 컨벤션 문서**로 관리하되:

1. `#` 명령어로 지속적으로 업데이트
2. 불필요한 내용은 쳐내어 가볍게 유지
3. "왜(Why)"를 포함해서 작성
4. 용도에 따라 파일 계층 분리
5. 작업 프로세스까지 정의하여 AI 행동 제어

이렇게 하면 시간이 갈수록 AI가 프로젝트를 더 잘 이해하고, 여러분이 원하는 방식대로 작동합니다.

</Callout>
